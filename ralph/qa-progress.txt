# Ralph QA Progress — GEM India

## QA Patterns

<!-- Cross-feature QA findings: common bug classes, edge cases worth re-checking. Codex writes here. -->
- Event-scoped API response-shape QA should compare visible unit expectations against the frozen contract text; stale route tests can pass while a contract like CE14 expects a singular row/null shape and reads properties directly. [UPDATE 2026-04-16: eventid-scoping CE14 bumped to v3 — travel is now an array of legs (inbound/outbound/ground/rebookings). Original v2 singular-shape expectation was a contract-authoring gap, not a production bug. Check contract version before flagging shape mismatches.]
- Read-only UI QA should check every mutation control on the surface, including secondary lifecycle controls like red-flag review/resolve buttons, not just primary Add/Create buttons.
- Bulk generation QA must follow the lock across the async worker boundary: route-level SET NX is insufficient unless the queued worker receives the same lock key and deletes it in success and failure paths.
- Certificate Playwright contracts are currently blocked before product assertions by Clerk auth setup redirecting `/events` with `ERR_TOO_MANY_REDIRECTS`; record this as a harness blocker when it recurs rather than treating it as certificate product behavior.
- Repo-level `npx tsc --noEmit` currently fails on broad existing errors across QA/test, Sentry, people, navigation, notification, and cascade files; distinguish those from story-local production type regressions.
- Certificate template activation can pass archive-before-activate ordering while still doing extra no-op archive writes; explicitly check the no-sibling active-template path against the PRD.
- Stryker baseline story cert-code-005 is accepted only under outcome b: verify the committed snapshot has >=60 scored modules, a plain-English partial-baseline note, preserved travel scores, and no `stryker.conf.mjs` shadow config in the working tree.
- API routes that call `assertEventAccess(eventId, { requireWrite: true })` can accidentally collapse `EventArchivedError` into a generic 403 catch-all; archived-event contract checks need a separate 400 assertion.
- Backend event-scoped POST routes can validate the main body while silently stripping submitted `eventId`/`event_id`; explicitly probe path-vs-body event ID mismatches and Sentry mismatch capture.
- Probe both camelCase `eventId` and snake_case `event_id` mismatch bodies; a route can reject one while Zod strips the other before the mismatch helper sees it.
- Certificate rendered-variable snapshots must be resolved from authoritative DB state at issuance time, not trusted from caller-provided variables; stale master-person values become permanent once inserted.
- Bulk APIs that return `batch_id` need a durable retrievable summary keyed by that id; an Inngest handler return value alone is not an operator-visible batch report.
- Public certificate verification must satisfy the frozen `/verify/{token}` contract, not only `/api/verify/{token}`; also verify counter persistence is awaited and any follow-up cert lookup is event-scoped.
- Event-scoped routes outside the currently focused module can still catch `EventNotFoundError` as generic forbidden; audit every `assertEventAccess()` route for `crossEvent404Response()` before accepting CE4-style no-leak stories.
- Auth/helper modules imported by server components must not use the `'use server'` directive when they export classes/constants; Next rejects non-async exports at runtime, and Vitest may not catch it.
- CE14-style shared-person fixtures can be sparse: a person may belong to Event A and B while current test data has zero travel/session/registration rows in both. When that happens, manual QA should still verify the live route’s empty-array/null branch against the DB and separately probe the missing-`event_people` 404 path, rather than assuming the fixture is rich enough to prove leakage.
- Cascade notification idempotency QA should inspect the actual key stored/checked by `sendNotification`, not only handler-level arguments; legacy `notify:*` keys with timestamps can pass visible handler tests while violating the frozen `notification:{userId}:{eventId}:{type}:{triggerId}:{channel}` contract.
- Notification log retry QA must verify the send orchestration uses the upsert path, not only that a standalone helper has ON CONFLICT semantics; helper-only tests can pass while `sendNotification` still does plain inserts.
