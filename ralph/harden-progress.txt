# Ralph Harden Progress Log

## Hardening Patterns

- **UUID-format ID assertion**: When a function builds a deterministic hash-based UUID (v5 pattern),
  assert the `id` field in the insert call matches
  `/^[0-9a-f]{8}-[0-9a-f]{4}-5[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/`.
  This kills BlockStatement, ArrayDeclaration, MethodExpression (slice→hash), join-separator,
  and segment-prefix mutants in one test.

- **Exact UUID determinism assertion**: To kill StringLiteral and LogicalOperator mutants inside
  a UUID-building function (e.g. fallback string `'event'` → `''`, or `??` → `&&` on hash nibble),
  recompute the expected UUID in the test using the same algorithm with `createHash` and assert
  `insertChain.values` was called with `{ id: expectedId }` (exact equality, not just regex).
  This pins the exact hash input and variant nibble, killing mutants that the regex test cannot.

- **SessionId determinism test**: For hash IDs that incorporate sessionId, add a test with TWO
  DIFFERENT non-null sessionIds and assert `insertedIds[0] !== insertedIds[1]`. A single
  null-vs-uuid test is NOT enough — the `??→&&` mutant still produces different hashes for
  null vs a uuid. But two non-null UUIDs collapse to the same hash under `&&`.

- **isDuplicateError coverage requires three paths**: (1) message-only duplicate (no code),
  (2) code-only duplicate (no 'duplicate key' in message), (3) non-Error object.
  Without all three, the OR→AND, toLowerCase→toUpperCase, and instanceof mutations all survive.

- **assertEventAccess args**: Always assert `mockAssertEventAccess.toHaveBeenCalledWith(eventId, { requireWrite: true })`
  in at least one success-path test. Omitting this lets ObjectLiteral `{}` and BooleanLiteral `false` mutants survive.

- **DB select clause ObjectLiteral mutants** (e.g. `.select({...})→.select({})`) are **equivalent mutants**
  in mocked DB test contexts — the mock returns pre-set data regardless of the select shape.
  Note them as equivalent rather than spending iterations trying to kill them.

- **"inner vs outer catch" boundary**: If a function has an outer try/catch wrapping an inner
  try/catch, the `!registration` (not-found) check mutant `→false` is NOT equivalent — the outer
  catch fires with a generic message. Add a message-content assertion (e.g. `toContain('not recognized')`)
  to distinguish the two paths.

- **personName in result**: Verify `result.results[0].result.personName` is the actual DB name
  (not 'Unknown') to kill the `??→&&` mutant that collapses any truthy name to 'Unknown'.
  Also add a test where the person query returns `[]` to verify the optional-chaining fallback
  produces a synced record (not an error) with personName 'Unknown'.

- **Optional-field null coalescing**: For `field || null` patterns in DB inserts, add TWO tests: one with the field absent (expect null) and one with a value provided (expect the actual value). This kills both `||→&&` (LogicalOperator) and `ConditionalExpression→true/false` mutants in one pair.

- **Slug prefix extraction**: To test slugify truncation/separator behavior through a higher-level function, strip the timestamp suffix with `slug.slice(0, slug.lastIndexOf('-'))` since base36 timestamps contain no dashes.

- **Terminal-state transition error message**: For event status machine tests, assert the error message contains `'none (terminal state)'` when transitioning from archived/cancelled. This kills StringLiteral, ConditionalExpression, and LogicalOperator mutants on the fallback string in one assertion.

- **Slug leading/trailing dash strip**: To kill the StringLiteral mutant on `.replace(/(^-|-$)/g, '')` ('' → "Stryker was here!"), use a name like `'---hello world---'` and assert the slug prefix equals `'hello-world'` with no leading/trailing dashes.

- **Delegate-arg ObjectLiteral kill**: For server actions that call a service function with `{eventId, notificationLogId, initiatedByUserId}`, assert the exact call args with `toHaveBeenCalledWith({...})`. This kills ObjectLiteral `{}` mutants on the args object. Also assert `assertEventAccess` was called with `{ requireWrite: true }` — this kills both ObjectLiteral and BooleanLiteral mutants on the options object.

- **Schema mutants in 'use server' files**: When a module has `'use server'`, the local-replica schema in the existing test file only tests the replica, not the actual source schema. To kill schema mutants in the source, call the actual exported function with invalid inputs (e.g. non-UUID eventId, invalid enum value, out-of-range limit/offset) and expect them to throw. The mutated schema (e.g. `z.object({})`) will silently accept the invalid input, causing the function to proceed and the test to fail (kills the mutant).

- **Enum boundary pair**: To kill both ArrayDeclaration (`[]`) and StringLiteral (`""`) mutants on a z.enum, add BOTH a success test (valid enum value succeeds) AND a rejection test (invalid value throws). The empty-array mutant is killed by the success test; the wrong-string mutant is killed by the rejection test.

- **MethodExpression min→max kill**: For `z.number().int().min(N)` mutant to `max(N)`, test with `N+1` — it passes `min(N)` but fails `max(N)`. For offset `min(0)`, test `offset=1`.

- **every→some kill requires a "partial match" array/object**: To kill `a.every(...)→a.some(...)`, craft inputs where at least one element matches but at least one doesn't. For arrays: `['a','b']` vs `['a','c']` — `'a'` matches so `some` returns true while `every` correctly returns false. For objects: `{a:1,b:2}` vs `{a:1,b:3}` — key `'a'` matches so `some` would return true. The existing order-mismatch or length-mismatch tests don't reach L45/L53.

- **Primitive inequality to kill L47→true object-branch**: When deepEqual is mutated to always enter the object branch (condition→true), `Object.keys(1)=[]`, `Object.keys(true)=[]` — vacuously equal. Add `evaluateGuard({count:1},{count:2})→false` and `evaluateGuard({flag:true},{flag:false})→false` to catch this.

- **null-value NoCoverage**: deepEqual's null-check branch (L41) only fires when a payload leaf value OR a guard value is `null`. Tests comparing nested objects never produce null deepEqual args unless you explicitly use `{x:null}` in the guard or payload.

- **sanitizeKeyField exact-value assertion**: The existing colon-injection test only checked `key1 !== key2`. That inequality holds even when `:` is removed entirely (`''` mutant). Add an exact-value assertion like `toContain('ev_001')` to pin the replacement to `'_'`.

- **S3Client config ObjectLiteral/StringLiteral mutants**: In a mocked AWS SDK context, the mock ignores constructor args. Assert `MockS3Client.toHaveBeenCalledWith({ region: 'auto', endpoint, credentials: { accessKeyId, secretAccessKey } })` after each operation call. This kills region StringLiteral, outer config ObjectLiteral, and credentials ObjectLiteral in one assertion per method.

- **async catch block NoCoverage via constructor throw**: If a function's `try` block has no `await` on the final return (e.g. `return presigner(...)`), async rejections bypass `catch`. To cover the catch block, throw synchronously from the constructor mock (`MockS3Client.mockImplementationOnce(() => { throw err; })`). For operations WITH `await` (upload, delete), `mockSend.mockRejectedValueOnce(err)` reaches the catch block normally.

- **Hex checksum format test**: Assert `result.fileChecksumSha256.toMatch(/^[0-9a-f]{64}$/)` to kill both the join-delimiter StringLiteral (`''` → `"Stryker was here!"`, makes output much longer) and the padStart pad-char StringLiteral (`'0'` → `''`, causes short bytes to lose leading zero and total length < 64).

- **ExcelJS fill/font assertions**: To kill HEADER_FILL/HEADER_FONT ObjectLiteral/StringLiteral/BooleanLiteral mutants, parse the generated Excel buffer with ExcelJS and assert `cell.fill.type === 'pattern'`, `fill.fgColor.argb === 'FF1F4E79'`, `font.bold === true`, `font.color.argb === 'FFFFFFFF'`, `font.size === 11`. ExcelJS fully round-trips styling through the .xlsx binary.

- **ExcelJS cell value precision**: Reading `ws.getRow(N).getCell(M).value` from a parsed buffer kills ArrayDeclaration/BlockStatement mutants on `ws.addRow([...])`. ExcelJS returns null (not '') for cells that were never set, and '' for cells explicitly set to empty string — distinguish these when the test cares.

- **CSV header exact-match**: `expect(csv.split('\n')[0]).toBe(expectedHeaderString)` kills ALL column-name StringLiteral mutants in one assertion. Never use `.toContain()` per column — it leaves StringLiteral('' → mutant) alive.

- **sanitizeFileName test via generateEventArchive**: The helper function is private; test it by feeding fileName strings through generateEventArchive's cert entry, then reading the appended name from archiver._entries. Use a per-test helper `runWithFileName(name)`.

- **Filename dedup NoCoverage**: Pass ≥2 certs with identical fileName to generateEventArchive and assert the second gets a `-2` suffix (and third gets `-3`). This covers the entire L308-316 dedup block that is otherwise NoCoverage.

- **uploadStream NoCoverage**: Create a storage stub with an `uploadStream` method; generateEventArchive takes the if-branch and never calls `upload`. Assert `uploadStream` called once, `upload` not called.

- **?? fallback StringLiteral kills require asserting the non-null value too**: For `field ?? ''` mutants, you need BOTH a null test (parts[N]='' kills StringLiteral `''`→"Stryker") AND a non-null test (parts[N]='ActualValue' kills LogicalOperator `??`→`&&`). Without the non-null assertion, `??→&&` survives because `truthy && ''` = `''` which looks the same as the original when you only check the null case.

To cover it, ensure the operand IS null in at least one test.
