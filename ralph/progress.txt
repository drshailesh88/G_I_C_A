# Ralph Progress Log — GEM India

## Codebase Patterns

<!-- Reusable patterns discovered during builds. Ralph writes here. -->

- issueCertificate retry loop must use db.transaction() — tests mock mockDb.transaction and count calls
- Supersession updates must be a single .set() call with status, supersededById, and updatedAt together (after insert, not before)
- Person/template validation stays OUTSIDE the transaction retry loop; only the cert-number-dependent logic goes inside
- Transaction-level retry uses PG error class 40 (code.startsWith('40')) to avoid conflicting with class-23 collision/constraint tests
- Stryker baseline with locked stryker.config.json (no ignoreStatic) cannot complete on laptop hardware — static mutants (26% of total, ~2612 mutants) dominate runtime and exceed the 60-min controller timeout. Shadow config files are LOCKED; partial baseline (72/102 scored modules) is the acceptable outcome per the PRD's outcome ladder.
- Controller's parseStrykerReport uses (killed+survived+timeout) as denominator — files with only NoCoverage or Ignored mutants get score=null and are skipped by applyMutationMeasurement
- API route pattern: type Params = Promise<{ eventId: string }>; await params to destructure; paramsSchema validates UUID; assertEventAccess for auth; safeParse body with Zod; catch errors and map to status codes per contract
- Certificate endpoints require super_admin or event_coordinator (not ops) — need extra role check beyond assertEventAccess's WRITE_ROLES which includes ops
- API request bodies use snake_case per contract; Zod .transform() maps to camelCase for internal action functions
- Bulk endpoints with Redis locking: use SET NX EX with JSON metadata (lock_holder, started_at); store lockKey in Inngest payload so handler can DEL on completion; mock @upstash/redis constructor in tests
- Bulk generation uses per-cert atomicity: each recipient gets its own try/catch + db.transaction(); failures reported in errors array, not thrown; no intermediate/pending status rows
- CME validation uses createCmeVariablesSchema(maxHours) factory — event duration fetched in route handler, schema created at request time with runtime max; non-CME types skip this validation

## 2026-04-15T12:40:00Z — cert-api-010 — POST bulk with Redis lock
- Implemented: Created POST /api/events/[eventId]/certificates/bulk route with Upstash Redis distributed lock. Lock key format: lock:certificates:generate:{eventId}:{type} with 5-min TTL via SET NX EX. Stores JSON metadata (lock_holder, started_at) in lock value. Returns 409 with holder info when lock held, 202 { batch_id, total, lock_expires_at } on success. Lock released via DEL on handler exception. Enqueues to Inngest for async bulk generation.
- Files changed: src/app/api/events/[eventId]/certificates/bulk/route.ts (new), route.test.ts (new)
- Tests added: 7 (lock acquired with NX+TTL, 409 when held, lock released on exception, 403 ops role, 400 invalid params, 202 response shape, 400 invalid body)
- Learnings: Mock @upstash/redis by mocking the Redis constructor to return an object with mock set/get/ttl/del methods. Lock value stored as JSON.stringify so Upstash auto-parses on GET.
