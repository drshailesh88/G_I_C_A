## Attack Patterns

<!-- Recurring attack vectors + their fixes. Agent writes here. -->
- Code-seeded registries must be deeply frozen before export. Mutable module-level arrays and nested lists let one runtime caller poison later seed/render behavior for the whole process and can even grow the shared registry with forged entries.
- Runtime lookup registries used for `key in registry` validation must use null-prototype, immutable objects. Plain object registries inherit `__proto__`, `constructor`, and `toString`, which lets malformed route params pass validation and lets one runtime caller poison metadata for later requests.
- Redis-backed provider/state helpers must exact-match a small allowlist before building keys. Whitespace-padded, unknown, or oversized provider strings otherwise create parallel key namespaces, bypass open-circuit state, and spray unbounded Redis keys.
- Webhook correlation lookups must scope provider message IDs by provider, not message ID alone. The schema only guarantees uniqueness for `(provider, providerMessageId)`, so same-channel providers that emit the same upstream ID can otherwise advance the wrong notification log.
- Notification provider adapters are security boundaries too: validate event-scoped attachment storage keys and restrict outbound display names/custom headers before building provider payloads, or direct callers can sign cross-event private files and inject provider-level email headers.
- CSV import helpers are security boundaries too: reject dangerous object-prototype header names, fail closed on mapped `__proto__`/`constructor` lookups, and block spreadsheet-formula payloads before imported text can crash previews or become stored export-time formula injection.
- Redis-backed feature-flag helpers are security boundaries too: validate canonical `eventId` segments and runtime flag keys before building keys, or malformed tenant IDs will miss to default-open flags and prototype-chain/default-registry lookups can return truthy non-booleans.
- Storage helpers must validate object-key canonical form and signed-URL TTLs before touching the S3/R2 SDK. Otherwise traversal-style keys or overlong expiries can bypass higher-level event scoping assumptions and turn private files into long-lived download URLs.
- Shared event-scope SQL helpers must validate canonical UUID event IDs, not just truthiness. Otherwise malformed, whitespace-padded, or oversized tenant IDs reach UUID-backed queries and turn missing caller-side validation into DB-layer failures instead of a fail-closed scope guard.
- Export helpers are still security boundaries even when API routes validate first. Validate canonical event IDs inside the helper itself before building storage keys or issuing UUID-backed queries, and event-scope every joined event-owned table in export reads to stop legacy foreign keys from leaking cross-event metadata.
- Excel/XLSX export joins must event-scope every foreign event-owned table, not just the base scoped rows. Bare joins from scoped assignments/attendance into `sessions`, `halls`, or `event_registrations` still leak cross-event metadata when bad legacy foreign keys point outside the active event.
- Idempotency-key builders must encode and presence-mark each segment. Raw delimiter concatenation and placeholder sentinels let malformed present values collide with missing fields or adjacent segments, suppressing distinct notifications.
- Notification-log idempotency conflicts must still be tenant-scoped. A globally unique `idempotency_key` is not enough on its own: `ON CONFLICT DO UPDATE` paths must require the same `eventId`, or a direct caller can mutate another event's log row or block the wrong tenant by reusing its key.
- Service orchestrators must reuse the hardened persistence helpers instead of carrying private SQL copies. Duplicated local begin/upsert logic drifts from the canonical fix and can silently reopen cross-event idempotency conflicts on the default production path.
- Logistics server actions cannot rely on UI-only filters. Enforce travel-first and event/person-scoped foreign keys inside the action or direct calls will create cross-tenant/invalid rows.
- Logistics mutations must emit audit + cascade side effects from the server action itself. If the action only mutates the table, downstream flags/notifications are bypassable.
- Fetch-then-update by bare `id` is unsafe in logistics flows. Include `eventId` and a compare-and-swap guard (`updatedAt` when available) in the write query to block stale concurrent writes.
- Read actions must validate the route `eventId` before auth/database access and reject body/route `eventId` mismatches, or malformed/stale requests can bypass Zod until after DB work starts.
- Logistics action modules cannot rely on generic event access alone. Travel data is ops-owned, so direct server-action calls need module-specific role allowlists or `org:event_coordinator` can bypass the UI and operate on logistics records.
- Event-scoped read joins must scope every joined event table, not just the base table. Joining by globally unique IDs alone can leak cross-event metadata from bad or legacy rows.
- Multi-query aggregate dashboards need a stable snapshot (`repeatable read` or a single aggregate query). A default Postgres transaction can still produce torn stats under concurrent writes.
- Action-generated workspace hrefs can silently rot after route refactors. Treat dashboard/action-returned links as production behavior and lock them to live route paths with regression tests.
- Attendance/check-in mutations must event-scope optional `sessionId` foreign keys before inserting attendance rows, or offline/manual flows can attach active-event attendance to another event's session.
- Session-aware check-in reads must validate optional `sessionId` parameters for UUID shape and active-event membership before using them to shape attendance state, or stale/foreign session IDs can bypass validation and mislead operators.
- Check-in write actions must validate the route `eventId` before auth/database work, reject body/route `eventId` mismatches, and event-scope optional `sessionId`s before duplicate detection or inserts. Otherwise direct calls can create or probe attendance under stale or foreign scopes.
- Branding and file-backed settings must event-scope stored object keys before signing URLs; direct server-action JSON updates can otherwise borrow another event's object.
- Server actions must validate runtime enum parameters and uploaded file bytes. TypeScript unions and browser-provided MIME strings do not protect direct action callers.
- JSON blob read-merge-write actions need compare-and-swap guards on `updatedAt`, or concurrent partial updates can silently clobber unrelated fields.
- Long-running bulk jobs cannot ignore distributed-lock renew failures. If the lock expires or ownership is lost mid-flight, abort instead of continuing duplicate work without exclusivity.
- Async queue entrypoints must validate event-scoped resource IDs before dispatch. If they trust the worker to filter later, cross-event or stale IDs still queue misleading background jobs.
- Feature-flag checks for destructive bulk actions must fail closed on provider/Redis errors. Swallowing lookup failures turns an outage into a gate bypass.
- Certificate issuance must bind active templates and polymorphic eligibility basis IDs to the requested event/person/type. Admin override does not mean arbitrary cross-event backing records or mismatched template snapshots are safe.
- Certificate supersession must clear the old current row with a status-scoped compare-and-swap before inserting the replacement, or the partial one-current unique index and concurrent regenerations can break the chain.
- Notification idempotency keys for certificate sends must be stable per certificate/channel. Timestamped keys convert rapid duplicate clicks into distinct sends and bypass Redis dedupe.
- Template editor actions must validate merged allowed/required variable sets on partial updates and use compare-and-swap guards on status/updatedAt during update/activate/archive, or stale direct calls can persist invalid templates or revive old state.
- Global admin actions that do not yet have an `eventId` still need explicit Clerk role enforcement. Checking only `userId` lets `org:ops` or `org:read_only` bypass UI-disabled writes and create top-level resources directly.
- Module-specific communications actions cannot rely on generic `assertEventAccess(..., { requireWrite: true })`. `org:ops` inherits broad event write access there, so communications reads/writes need their own role allowlists to block PII leakage and resend control bypasses.
- People-directory actions need their own module RBAC even when they are not event-scoped. Bare `auth().userId` checks let `org:ops` mutate the master people table, and event-roster helpers still need UUID validation plus event access before reading or forging `event_people` links.
- Program actions cannot rely on generic event write access. Halls, sessions, assignments, invites, versioning, and schedule reads need program-specific role allowlists or `org:ops` can bypass the UI and operate on scientific-program data directly.
- Cascade handlers must resolve authoritative event-scoped resource, person, and event context before trusting payload `personId` or snapshot `variables`. Forged envelopes can otherwise redirect notifications, mis-target downstream flags, and omit required template vars like `fullName` and `eventName`.
- Certificate distributed-lock helpers must canonicalize and validate scope segments and bound TTLs. Raw key concatenation or unbounded expiries let malformed inputs bypass exclusivity, collide unrelated locks, or strand stale certificate locks after crashes.
- Certificate numbering helpers must validate the prefix format and enforce the 99,999 sequence ceiling before planning or issuing. Raw regex prefixes can borrow other type numbers, and unbounded incrementing can emit impossible six-digit certificate IDs that jam issuance paths.
- Client-side offline queues must validate and normalize rows before persistence and while reading back legacy data. One malformed local record can poison a whole server-validated sync batch and deadlock offline recovery until storage is cleared.
- IndexedDB queue state should use an explicit numeric/string sync key and cursor-based reads for cleanup/counting. Mixed boolean/numeric lookups and `getAll` on unbounded queues can hide pending rows or turn badge polling into a local memory sink.
- Session and invite lifecycle writes need optimistic-lock filters on `updatedAt` plus current status. Fetch-then-update by bare id lets stale editors or racing responders overwrite newer program state.
- Program assignment/invite creation must reject archived or anonymized people. UUID shape plus a valid FK is not enough when the people record has been retired from active use.
- Registration actions cannot rely on generic event access. Without module-specific role allowlists, `org:ops` can read attendee PII and mutate registration status directly.
- Public flag gates must fail closed when the provider/backend errors. Swallowing `registration_open` lookup failures silently reopens registration during outages.
- Registration status transitions must enforce access before the row lookup and update with compare-and-swap filters on `eventId`/`status`/`updatedAt`, or unauthorized callers can probe existence and stale editors can clobber newer state.
- Clerk organization-membership invariants cannot trust a single 100-row page or a bare read-before-write. Paginate membership queries fully and serialize destructive org membership mutations, or large orgs and concurrent admin changes can strip the last super admin.
- Transport assignment actions must event-scope both the `travelRecordId`/`personId` pair and the target `vehicleAssignmentId`/`batchId` pair. UUID-shaped foreign keys alone let direct calls graft passengers onto another event’s travel or the wrong batch.
- Cascade emitters must validate event-specific payload shape and any duplicated tenant identifiers before enqueueing background work. Trusting downstream handlers lets malformed or cross-tenant events pollute Inngest/captured-event state before they fail.
- Optional enrichment at emit time must fail open. If snapshot/contact lookups throw before enqueue, the durable cascade side effects are lost even though the base event was otherwise valid.
- QR payload helpers must enforce the same length budget and canonical URL shape as the attendance validators. If they trust URL normalization or unbounded base URLs, malformed `../checkin`/duplicate-param payloads and self-invalidating oversized QR codes slip through.
- Shared event auth helpers must validate UUID event scopes themselves and require a real event row even for super-admin bypasses. Trusting callers or bypassing existence checks lets malformed or phantom event IDs reach downstream auth/DB code.
- Automation payload sanitizers must recurse through nested objects and arrays. Top-level-only filtering leaks tokens/passwords into template variables and persisted notification logs.
- Cascade notification handlers cannot trust `payload.variables` for recipient routing or template identity fields. Rebuild recipient/event/fullName vars from scoped DB state before dispatch, or forged snapshots can redirect notifications and normal saved events can arrive without a valid destination/template context.
- Red-flag lifecycle writes need compare-and-swap guards on `eventId`/`flagStatus`/`updatedAt`. Reading a flag under scope and then updating by bare `id` lets stale reviewers overwrite newer review/resolve transitions.
- ZIP/file-export filename sanitizers must strip control characters, trim before removing leading dots, collapse `..`, and cap entry-name length. Otherwise whitespace-padded traversal names, null bytes, or oversized filenames survive into archive headers.
- Shared certificate/template registries must be exported as immutable data. Returning live config objects lets one caller mutate required variables or prefixes and poison later requests handled by the same server process.
- Fixed-format identifier helpers must validate both namespace membership and numeric bounds. Accepting unknown certificate prefixes or overflowed/fractional sequence numbers creates forged or impossible IDs that downstream code may treat as legitimate.
- Background workers and queue consumers are security boundaries too. Validate queued payload enums and tenant IDs, cap batch sizes, refuse forged cleanup lock keys, and stream large archive uploads instead of buffering whole artifacts in memory.
- Global seeders that create `eventId = null` defaults cannot rely on a partial `(event_id, channel, key)` unique index plus read-then-insert loops. Postgres permits multiple `NULL` keys there, so concurrent seed runs need transaction-scoped locking and atomic rollback or they will duplicate global rows and leave half-seeded state on failure.
